Supported Algorithms

  • EDF (Earliest Deadline First): Dynamic priority — always picks the ready task with the closest absolute deadline. Preemptive.

RTOS Scheduling — Literature Review

1. Introduction

1.1 Terminology

  • Task: A process in the Real Time environment
  • Active Task: A task that can be executed by the CPU
  • Ready Task: A task waiting in the queue for the CPU
  • Running Task: A task currently executed by the CPU
  • Scheduling Algorithm: Algorithm that defines task execution order

1.2 Scheduling

Given a set of tasks J = {J1, …, Jn}, a schedule is an assignment of tasks to the processor so that each task is executed until completion.

1.3 Properties

  • Schedulability: A task set is schedulable if there exists at least one algorithm that can produce a feasible schedule.
  • Feasibility: A schedule is feasible if all tasks complete according to their specified constraints.

1.4 Task Characterization

Given task Ji:

  • Parameters
    • ai: Arrival time
    • Ti: Period
    • Ci: Computation time (WCET)
    • di: Deadline
    • si: Start time
    • fi: Finish time
  • Metrics
    • Response Time Ri = fi − ai
    • Lateness Li = fi − di
    • Slack Time Xi = di − ai − Ci

1.5 Constraints

  • Timing: tasks finish before their deadline. Missed deadlines are classified as Hard, Soft, or Firm.
  • Precedence: tasks follow a DAG execution order.
  • Resource: tasks can only access unlocked resources.

1.5.1 Task Activation

  • Periodic Task: activated every period Ti
  • Aperiodic Task: activation time is unknown and irregular
  • Sporadic Task: aperiodic but with a minimum inter-arrival time

1.5.2 Precedence Constraints

Execution order is expressed with a DAG. Notation: Ja → Jb means Ja is an immediate predecessor of Jb.

1.5.3 Resource Constraints

Resource access is particularly critical in RTOS; classical semaphore-based locking is generally insufficient.

2. Scheduling Problem

2.1 Problem Definition

Given a task set J, processor set P, resource set R, and a DAG of precedence constraints, scheduling assigns processors and resources to tasks under all constraints.

2.2 Scheduler Classification

  • Preemption: Preemptive vs. Non-preemptive
  • Priority assignment: Static vs. Dynamic
  • Decision timing: Off-line vs. On-line
  • Optimality: Optimal vs. Heuristic

2.3 Scheduling Metrics

  • Average response time: t̅r = (1/n) ∑ (fi − ai)
  • Total completion time: tc = max(fi) − min(ai)
  • Maximum lateness: Lmax = max(fi − di)
  • Number of late tasks: Nlate = ∑ miss(fi)

2.4 Scheduling Anomalies

Graham (1966) demonstrated that optimising individual parameters (more processors, shorter computation, relaxed precedence) can paradoxically make a previously feasible schedule infeasible.

2.5 Problem Classification (Graham et al., 1979)

Three-field notation (α, β, γ): machine environment, job description, objective criterion.

3.1 EDD — Jackson 1955

Solves 1|sync|Lmax: sort tasks by deadline, execute in that order. O(n log n). Optimal for minimizing maximum lateness when all tasks arrive simultaneously.

3.2 EDF — Horn 1974

Solves 1|preem|Lmax: always run the ready task with the earliest absolute deadline. O(n²). Optimal for preemptive, asynchronous, independent tasks.

3.3 LDF — Lawler 1973

Solves 1|(prec,sync)|Lmax: schedule tasks in reverse deadline order, respecting the DAG. O(n²). Optimal for synchronous tasks with precedence constraints.

3.4 EDF* — Chetto 1990

Solves 1|(prec,async)|Lmax: transforms a DAG-constrained task set into an equivalent independent set by adjusting arrival and deadline parameters, then runs standard EDF.

4. Periodic Task Scheduling

di = Ti di ≤ Ti
Static Priority RMA — U ≤ n(21/n−1) DMA — Ri ≤ Di
Dynamic Priority EDF — U ≤ 1 PDA-EDF — Processor demand analysis

4.1 Schedulability Tests

Utilization factor U = ∑ Ci/Ti. Sufficient (not necessary) test: U ≤ Ulub(A).

Worst-case response time: Ri = Ci + ∑j∈hp(i) ⌈Ri/Tj⌉ Cj. Necessary and sufficient.

4.2 RM Algorithm — Liu & Layland 1973

Assign fixed priorities by period: shortest period = highest priority. Ulub = n(21/n−1) → 69.3% as n→∞.

4.3 EDF (Periodic) — Liu & Layland 1973

Dynamic priority based on absolute deadline. Ulub = 1. Necessary and sufficient schedulability condition.

4.4 DM Algorithm — Leung & Whitehead 1982

Static priorities assigned by relative deadline: shorter deadline = higher priority. Optimal when di ≤ Ti.

4.5 RM vs EDF Comparison

RM

  • Simple to implement
  • U < 69%
  • Under overload, lower-priority tasks are blocked predictably
  • Leaves slack for aperiodic tasks

EDF

  • More complex to implement
  • U ≈ 100%
  • Under overload, timing failures spread unpredictably

5. Mixed Task Scheduling

Three main approaches for periodic + aperiodic coexistence:

  • Immediate service: aperiodic tasks preempt periodic tasks on arrival
  • Background scheduling: periodic tasks are high-priority; aperiodic tasks run in idle time (FCFS)
  • Aperiodic server: a dedicated server task handles aperiodic jobs within its allocated bandwidth

5.1 Polling Server

At the start of each server period, sets capacity to Cs. Serves queued aperiodic tasks up to capacity; if queue is empty, capacity is discarded. Schedulability: Ups ≤ n[(2/(Us+1))1/n−1].

5.2 Deferrable Server

Like Polling Server but preserves unused capacity until end of period to serve late-arriving aperiodic tasks. More responsive, but tighter schedulability bound: Uds ≤ n[((Us+2)/(2Us+1))1/n−1].

5.3 Total Bandwidth Server (TBS)

Dynamic priority server using EDF. Assigns deadlines to aperiodic tasks so their bandwidth never exceeds Us: dk = max(rk, dk−1) + Ck/Us. Schedulability: Up + Us < 1.

Disclaimers & Simulation Notes

Use at your own risk — scheduling visualizations may contain inaccuracies.
  • Overhead: Context switching and scheduling computation time are assumed to be zero.
  • Time Limit: The simulation runs for 1 hyperperiod (LCM of all periodic task periods), capped at 200 ticks to prevent browser freezing.
  • Defaults: If a deadline is left blank for a periodic task, the relative deadline is assumed equal to the period.
  • Aperiodic deadlines: Entered values are treated as absolute deadlines.
